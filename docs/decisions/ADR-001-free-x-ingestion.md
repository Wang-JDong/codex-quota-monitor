# ADR-001：选择 RSSHub + X `auth_token` 作为免费入口

- 状态：已接受
- 日期：2026-07-13

## 背景

产品只需每小时读取四个账号的少量帖子。用户明确不接受为 X API 付费。同时，公共 RSS 代理会把认证信任、稳定性与请求记录交给第三方，不符合秘密边界。

## 决策

在 VPS 内按需启动项目私有 RSSHub，通过环境变量提供一个专用 X 会话的 `auth_token` Cookie，并仅在 loopback 端口提供 feed。RSSHub 版本由 `package-lock.json` 固定，Node.js 由项目私有运行时提供，不改动系统 Node.js。

免费适配器当前只使用 `UserTweets` 抓取四个白名单账号的顶层原创帖，并关闭回复和转发。转发、引用和回复不作为证据。已观测到 `UserTweetsAndReplies` 当前返回 HTTP 404，因此不声称支持回复监控；只有上游恢复且测试保护覆盖该路由后才能更改本决策。

## 理由

- 满足不使用付费 X API 的产品约束。
- Cookie 不经过第三方 RSS 服务，且 RSSHub 不向公网监听。
- JSON/RSS/Atom 让 Python 监控可以仅使用标准库，降低生产依赖。
- 按需生命周期适合低配 VPS，比常驻浏览器自动化更容易限制资源。

## 后果与风险

- 免费入口没有 SLA，X 或 RSSHub 改变可能使路由失效。
- Cookie 会过期，运维者必须定期轮换；不存储账号密码来自动重新登录。
- X 帐号应为专用、低权限会话；Cookie 泄露等同于会话泄露。
- 需要通过健康告警、dry-run 和原帖人工查看补偿上游不稳定性。

## 未选方案

- 付费 X API：语义更稳定，但直接违反成本约束。
- 公共 RSSHub 实例：维护更少，但无法控制日志、会话和可用性。
- 常驻浏览器：对页面变化脆弱，资源成本过高。
