# ADR-002：优先 at-most-once 的投递语义

- 状态：已接受
- 日期：2026-07-13

## 背景

向 Webhook 发送消息时，客户端在超时、断连、5xx 或无法解析的成功响应之后，不知道服务器是否已经接受消息。如果立即重试，用户可能收到重复的稀有事件告警；如果不重试，则可能漏掉一次投递。飞书自定义机器人不提供可由客户端控制的通用幂等键。

## 决策

为每个匹配帖子以及每个健康迁移（`alert` / `recovered`）持久化投递状态。发送前使用 SQLite 条件更新原子地将 `pending` claim 为 `in_flight`。健康迁移另存递增 epoch，使同一次故障的告警和恢复各有唯一身份。只有确认未被接受的 HTTP 429 可以自动重试并回到 `pending`。任何结果不确定的响应都进入 `uncertain`，不自动重发。

运维者必须先在飞书群查看消息是否已到达，然后对业务帖子使用 `delivery-resolve POST_ID --as sent|retry`，或对健康迁移使用 `health-resolve alert|recovered --as sent|retry` 做显式决定。

## 理由

- 本产品是低频、高信号通知，重复告警会快速损害信任。
- pre-send claim 保护并发运行和进程崩溃窗口；无论业务还是健康通知，重启后遗留的 `in_flight` 都会迁移为 `uncertain`。
- 人工核对频率很低，且比在没有服务端幂等支持的情况下猜测更可靠。

## 后果

- 不能数学上保证 exactly-once，但能避免系统在结果不确定时主动制造重复。
- 在运维者核对之前，`uncertain` 帖子或健康迁移不会再发，因此可能暂时漏投。
- `status` 必须显示 `pending`、`in_flight`、`uncertain`、`permanent_failed`、`sent` 和尝试次数。
- 不得通过直接编辑 SQLite 解决投递状态，以避免破坏状态约束。

## 未选方案

- 所有错误都自动重试：更接近 at-least-once，但会在最常见的不确定网络故障中制造重复。
- 不持久化投递状态：实现简单，但无法在重启和并发下审计或去重。
- 只依赖帖子 ID 内存去重：不能跨运行保护。
