# Codex Quota Monitor 产品需求文档

## 问题

Codex 额度重置或将要重置的消息可能先出现在 X。个人用户无法持续巡视，而全网关键词告警噪声很大。付费 X API 对单人、低频监控的成本不合理，同时自托管环境可能已承载 SSH、代理和订阅等关键服务，不能为一个通知工具承担网络或资源回归。

## 目标用户

主要用户是使用 Codex、愿意维护一台 Linux VPS、但不愿为 X API 付费的个人开发者。用户希望通过飞书获得低噪声通知，也愿意在会话过期或投递结果不确定时进行少量人工维护。

## Jobs to be Done

- 当可信的 OpenAI 官方账号或相关工作人员发布额度重置消息时，我想在一个检查周期内收到飞书通知，以便尽快恢复高优先级任务。
- 当没有足够可信信号时，我不想被通知；遇到可信但新颖的表达时，我可以接受明确标注为“可能是额度重置，请确认”的低置信度提醒。
- 当免费抓取链路失效时，我想只收到一次可行动的维护告警，而不是每小时被刷屏。
- 当飞书投递结果不确定时，我想核对后明确选择“已收到”或“重试”，以便避免自动重复通知。

## 使用场景

1. 日常静默：定时任务完成抓取和判定，没有匹配，不产生飞书消息。
2. 计划重置：白名单原创帖子明确说明 Codex 额度将在未来重置，系统发送一条带原文链接的通知。
3. 可保存重置次数：白名单原创帖子明确说明已向账户发放 `banked reset`，系统发送“可保存重置次数已发放”，不把它表述成额度已自动恢复。
4. 会话过期：RSSHub 明确报告认证失败，系统告警一次，用户轮换 Cookie；四来源全部恢复后通知一次。
5. 投递超时：帖子或健康通知记为 `uncertain`，不自动重发；用户查看飞书后使用 `delivery-resolve` 或 `health-resolve`。

## 功能需求

### FR-1 可信源抓取

- 系统必须只抓取 `@OpenAI`、`@OpenAIDevs`、`@thsottiaux` 和 `@sama`。
- 默认抓取周期为每 30 分钟一次。
- 抓取由本机私有 RSSHub 完成，不暴露对外监听端口。

### FR-2 保守判定

- 原文必须来自白名单原创帖，并满足产品 + 动作 +（额度限制或时间/恢复）证据门槛。
- 已完成、正在进行、未来计划、低置信度 `possible_reset` 和明确已发放的 `banked reset` 都是有效状态；低置信度状态必须在飞书卡片中显式标注，不得写成已确认恢复。
- `banked reset` 必须同时出现额度/用量上下文和明确发放语言，并使用独立状态 `banked_available`。
- 疑问、愿望、邀请、购买、否定、失败和周期性通用说明必须排除。
- 引用元数据（包括 `quoted_text`）仅用于审计和展示上下文，永不作为分类匹配证据；转发也不作为证据。
- 当前免费 X 适配器使用 `UserTweets` 读取四个白名单账号的顶层原创帖，并用 `UserMedia` 补充可能被主时间线漏掉的带媒体帖子（尤其是引用帖）；按 X status ID 去重，且每条记录都校验作者和 status URL。转发、引用正文和回复不作为证据；`UserTweetsAndReplies` 当前返回 HTTP 404，专门的回复监控必须等上游恢复，并在测试保护下验证后才可启用。

### FR-3 基线、去重与投递

- 每个来源首次抓取单独建立基线，首次基线静默。
- 同一帖子必须持久去重。
- 发送前必须原子 claim；只有 HTTP 429 且确认未被接受时才自动有界重试。
- 超时、断连、5xx 或无法确认的 200 必须记为 `uncertain`，不自动重发。

### FR-4 运维与健康

- 支持 dry-run、飞书测试、状态查询、资源校验、最近 7 天且最多 100 条历史未匹配帖子幂等重处理、人工投递核对和回滚。
- 四个来源连续三轮全部失败，或明确认证失败时，健康告警只发一次。
- 恢复通知只在全来源恢复后发送一次。
- 告警与恢复通知必须使用持久化 epoch 和发送前原子 claim；结果不确定时进入 `uncertain`，只能由 `health-resolve` 显式核对。

## 非功能需求

- 资源：生产单次任务不得突破 systemd `MemoryMax=384M` 与 `CPUQuota=30%`。
- 隔离：生产使用 loopback 端口 `1200`，dry-run 使用隔离的 `1201` 和临时 SQLite。
- 安全：秘密不进入 Git、命令行或日志；服务使用独立非登录用户。
- 兼容：生产监控仅依赖 Python 3.12 标准库；RSSHub 使用项目私有 Node.js。
- 可审计：规则、来源、投递状态与人工动作必须可在代码或 SQLite 中追溯。
- 安全变更：部署不得安装 Docker、升级系统、修改防火墙/SSH，不得停止或重启现有节点服务。

## 验收标准

1. 四个白名单来源在 dry-run 中均可读，且非白名单作者不能成为证据。
2. 分类测试覆盖四种额度状态及疑问、否定、转发、周期机制、推荐邀请和不可信引用的负例。
3. 首次生产运行只建基线；第二次无新帖子时完全静默。
4. 同一匹配帖子并发或重启后不会自动重复成功投递。
5. 429 可重试；业务帖子、健康告警和恢复通知在结果不确定时均为 `uncertain` 且不自动重发。
6. preflight 与 postflight 证明受保护服务、unit 哈希和监听者未变。
7. 全量 pytest、Python 编译、Shell 语法、文档链接和秘密扫描通过。

## 风险与边界

- X 可以随时改变登录、Cookie 或页面行为，免费入口没有 SLA。
- 白名单账号仍可能误述、被盗或被攻击；通知是信号，不是额度控制面确认。
- 确定性规则倾向低误报，因此可能漏掉新表达；规则变更必须补充测试样例。
- 不确定投递选择“不重复优先”，需要用户人工核对。
- 现有节点保护列表是部署环境契约，换机器前必须根据实际服务调整并复审。

## 路线图

- `0.1.x`：稳定单机部署、测试样例和故障说明。
- 候选：可配置的可信来源策略与更明确的规则版本迁移。
- 候选：除飞书外的通知适配器，但不改变默认静默和 at-most-once 语义。
- 不承诺：全网抓取、多租户 SaaS、付费 X API 或自动控制 Codex 额度。
